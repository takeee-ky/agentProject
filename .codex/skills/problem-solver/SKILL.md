---
name: problem-solver
description: 問題解決が必要なときに使う。答えるべきイシューの設定、有力仮説の立案、検証手順の特定までを行う。
---

# 目的

問題が曖昧な状態でも、まず答えるべきイシューを定め、有力仮説を先に置いたうえで、確認すべき事実と次の打ち手を短時間で整理できるようにする。

# 入力

- 困っている現象
- 期待している状態
- 分かっている制約
- 既に試したこと
- 必要なら関連ログやコード断片

# 手順

1. 現象、期待値、制約を分けて書き出す。
2. いま本当に答えるべきイシューを 1 つの問いとして定義する。
3. 未確認の前提と確認済みの事実を切り分ける。
4. そのイシューに対する有力仮説を 1 から 3 個に絞る。
5. 各仮説について、正しければ何が観測されるかを明示する。
6. 各仮説を最短で反証または支持できる確認手順を決める。
7. 仮説の確からしさと検証コストを踏まえて実行順に並べる。
8. 必要なら、解決後に再発防止の観点を 1 つだけ添える。

# 出力

- 問題の要約
- いま答えるべきイシュー
- 確認済みの事実
- 有力仮説
- 仮説ごとの確認ポイント
- 次にやること

# 使う場面

- エラーの原因がまだ特定できていないとき
- 何から調べるべきか迷っているとき
- 調査の優先順位を整理したいとき

# 使わない場面

- 実装方針が既に固まっていて、そのままコード変更に進めるとき
- 単なる知識質問で、切り分けや検証計画が不要なとき
- 長い設計議論や複数テーマの整理を一度にやりたいとき

# 禁止事項

- 原因候補を無制限に広げない。
- イシューが曖昧なまま調査を始めない。
- 仮説を置かずに手当たり次第に調べない。
- 確認していないことを事実として扱わない。
- 調査計画と実装計画を混同しない。
